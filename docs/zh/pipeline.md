# 翻译流程

[English](../pipeline.md)

文译以“先建立全书理解，再按章稳定翻译”为默认策略。可以通过 `config.yaml` 关闭不需要的阶段，以降低成本或缩短时间。

```text
读取输入
-> 解析章节、正文段落和 EPUB 目录
-> 识别源语言或使用指定语言
-> 预扫整书，生成章节梗概和全书概览
-> 分析样章，建立初始术语表和风格指南
-> 按章、按批初译
-> 实时抽取和更新术语
-> 可选对照原文精修与标点规范化
-> 使用最终术语库执行全书最终审校
-> 可选全书一致性 QA
-> 生成报告
-> 回填并组装指定格式的成品
```

## 全书理解与上下文

预扫会生成逐章梗概和全书概览。翻译每个批次时，提示词会按稳定信息优先的顺序提供角色风格、全书概览、章节梗概、相关术语表、最近译文和待译原文。

这使早期章节可以参考后续剧情，同时让同章相邻批次在代词、称谓、语气和跨段句意上保持衔接。

## 术语库

样章分析会建立初始术语库；翻译过程中会从已完成的源文和译文中抽取并更新人物、地名、组织、术语、招式和称谓。后续批次只注入当前章节匹配到的词条，避免无关术语占用上下文。

术语表用于约束后续翻译和最终审校，但它不能自动保证已经完成的历史译文全部回写一致。可通过 `glossary list` 和 `glossary conflicts` 检查条目与冲突，再结合审校、QA、报告或人工调整。

## 质量控制

- **段数对齐**：模型必须返回与输入等长的 JSON 数组；失败会重试，仍失败时逐段兜底。
- **精修**：可由独立的 `translation_llm` 先生成初译，再由主 `llm` 同时查看原文、初译、全书概览、本章梗概、相关术语和最近最终译文，修正误译并提升中文流畅度。按批串行执行，所以下一批读取的是上一批最终译文。
- **精修恢复**：主模型返回坏 JSON、空数组、段数不等或 provider 异常时递归拆分失败批次并重试；只有仍失败的单段才使用 `polish_fallback_llm`，失败原因与备用段号会持久化以便诊断。
- **内容策略回退**：主模型明确拒绝某批内容时先重试，再逐段定位；只有仍被拒绝的段落使用 `translation_llm` 精修，其余段落仍由主模型处理。
- **标点规范化**：统一中文全角句读；引号默认跟随源文，日文小说保留 `「」『』`，也可配置为大陆式 `“”‘’`。
- **最终审校**：全部章节翻译完成后才开始。开启自动裁定时，主模型先结合候选译名和局部原译文上下文解决术语冲突；随后每章从最终术语库提取相关术语快照，按连续文本块并行检查漏译、增译、误译、术语和人称。开启 `autofix_severe` 后，严重问题由发现问题的同一个 `review_llm` 根据审校意见定向修复；未配置独立审校模型时才复用主 `llm`。
- **结构化输出恢复**：JSON 无法解析时以全新请求追加更严格、简洁的格式提醒并重试；连续失败后递归拆分审校块或术语批次。术语原始批次的全部递归子批均验证成功前不会写库。
- **全书一致性 QA**：在收尾阶段检查术语、人称、语气和标点的一致性，默认只报告问题。

最终审校默认关闭。设置 `pipeline.review: true` 后，一键流程会在翻译与 QA 之间
插入最终审校；也可以将它作为独立、可断点续跑的阶段使用：

```bash
uv run trans-novel review book.epub
uv run trans-novel review book.epub --force
uv run trans-novel review book.epub --fix     # --no-fix 覆盖为不自动修复
```

即使关闭 `pipeline.review`，显式调用上述命令仍会执行审校。`--force` 会重新
检查当前译文已完成审校的章节；`--fix` 和 `--no-fix` 会在当次运行中覆盖
`pipeline.autofix_severe`；`--resolve-conflicts` 和 `--no-resolve-conflicts`
会覆盖术语冲突自动裁定开关。

## 断点续跑

每个完成批次都会立即写入状态目录。再次运行 `translate` 会跳过已有译文，仅补齐未完成部分；`assemble` 可以直接从状态目录重新导出成品。
